Architecture Reference

Factory Inventory Management System

Full-stack demo for inventory tracking, order management, demand forecasting, and spending analytics. Vue 3 frontend, FastAPI backend, in-memory mock data loaded from JSON at startup.

Vue 3 + Vite
  • Vue 3.4 (Composition API)
  • Vue Router 4.3
  • Axios 1.6 (HTTP client)
  • Vite 5.2 (dev server, port 3000)

System Architecture

Data Flow — Filter-driven request lifecycle

  1. User changes a filter FilterBar updates a ref in useFilters (Time Period, Warehouse, Category, Order Status).
  2. View reacts and calls api.js A watch() on the filter refs invokes the relevant api.getX(filters) method.
  3. api.js builds the query string Filter values that aren't 'all' are appended via URLSearchParams; axios issues GET /api/....
  4. FastAPI handler filters in memory apply_filters() chains warehouse/category/status, filter_by_month() handles months and quarters (Q1–Q4).
  5. Pydantic validates the response response_model=List[Model] guarantees the JSON shape; clients can rely on field types.
  6. View stores raw data; computed derives the rest Raw arrays live in refs (allOrders, inventoryItems); KPIs, charts, and tables are computed().
  7. Template re-renders Vue's reactivity updates only the dependent DOM nodes; SVG charts recompute via their computed() data sources.

Architectural Notes

Why no database

This is a workshop demo. State lives in Python lists initialised from JSON files at startup. Reads are fast; writes don't persist. Restart resets everything. Production deployment would need to add a real datastore, auth, pagination, and CORS lockdown.

Why one filter system

Four filters (Time Period, Warehouse, Category, Order Status) flow through every list endpoint via query params. The same shape feeds apply_filters() on the server, keeping the contract uniform across views.

Refs vs computed

Raw API responses are stored in ref(). Every derived value — KPIs, chart series, filtered tables — is a computed(). This avoids manual recomputation and keeps reactivity correct when filters change.

Single-file backend

main.py defines all 13 routes and their Pydantic models inline. The CLAUDE.md guidance suggests extracting models.py/services//utils/ if the project grows, but the current size doesn't justify the indirection.
